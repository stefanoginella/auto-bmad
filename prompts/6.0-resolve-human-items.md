## Resolution Categories

### 1. [bad_spec] and [intent_gap] Change Log Entries — Validate AI Decisions

These entries record decisions the pipeline already made. Your job is to VALIDATE them, not redo them.

For each [bad_spec] and [intent_gap] entry in the story Change Log:
1. Read the referenced upstream artifact (PRD, architecture doc, epic)
2. Check: does the decision align with the upstream artifact's intent?
3. If YES — the decision is sound:
   - Add `[ai-resolved]` tag after the existing tag in the Change Log (e.g., `[bad_spec] [ai-resolved]`)
   - In the resolution report: record finding ID, the upstream evidence that supports the decision, and confidence level (high/medium)
4. If NO — the decision contradicts upstream intent:
   - Do NOT modify the Change Log entry
   - In the resolution report: mark as 🔴🔴 Critical, explain the contradiction, cite the specific upstream text
5. If UNCERTAIN — upstream artifacts are ambiguous or silent:
   - Do NOT modify the Change Log entry
   - In the resolution report: mark as 🔴🔴 Critical with "insufficient upstream evidence"

HARD RULE: Never fabricate upstream evidence. If the upstream artifact does not clearly support the decision, escalate to Critical. Do not guess, infer intent beyond what is written, or cite documents you have not read.

### 2. Unresolvable Findings from Triage Reports — Second Pass

Read the "## Unresolvable Findings" sections from both triage reports (if they exist).

For each unresolvable finding:
1. Re-read ALL upstream artifacts (PRD, architecture, epic, project context) with the full story implementation context now available
2. Search for information that was not available during the original triage pass (steps 1.4 / 3.5)
3. If NOW resolvable with concrete upstream evidence:
   - Apply the resolution to the story file (same rules as step 1.4 / 3.5 — add Change Log entry with appropriate tag)
   - Add `[ai-resolved]` tag to the Change Log entry
   - In the resolution report: record finding ID, what new evidence was found, and where
4. If STILL unresolvable:
   - In the resolution report: mark as 🔴🔴 Critical, restate what is missing

### 3. [defer] Change Log Entries — Verify Legitimacy

For each [defer] entry:
1. Check: is this item genuinely pre-existing (not caused by this story)?
2. Check: could the current story have addressed it within scope?
3. If genuinely pre-existing and out of scope: leave as-is, note "verified" in resolution report
4. If the current story could address it: flag in resolution report as "incorporation candidate" (do NOT auto-incorporate — just flag)

### 4. Manual Test Scenarios — Check Coverage

If the acceptance audit ({{ACCEPTANCE_REPORT}}) exists, check whether automated acceptance tests already cover scenarios that would otherwise require manual testing.

3. If genuinely requires human interaction (visual verification, physical device, etc.): leave as-is
